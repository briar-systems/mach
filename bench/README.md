# Compiler benchmark harness

A small, reproducible way to measure how fast the Mach compiler compiles
representative inputs, and to catch performance regressions over time.

## Running

```sh
bench/run.sh [path-to-mach] [iterations]
```

- `path-to-mach` — compiler binary to benchmark. Defaults to `out/bin/mach`
  (build it first with `make clean && make`).
- `iterations` — timed runs per input. Defaults to `5`, or `$BENCH_ITER`.

Pass an empty first argument to keep the default compiler while setting
iterations, e.g. `bench/run.sh "" 10`.

Only `bash` and coreutils are required.

## What it measures

End-to-end `mach build` wall-clock time for each input. Each input is built
`iterations` times from a clean object tree (a full rebuild every run, not an
incremental one), and the harness reports the **minimum** and **median** wall
time in milliseconds.

This is whole-pipeline time — config parse, load, sema, lower, optimise,
codegen, and link together. The compiler CLI does not expose per-phase timing,
so phases are not broken out; the `-v` flag only lists per-module stage
transitions, not durations.

### Inputs

| Input               | Workload                                                    |
| ------------------- | ----------------------------------------------------------- |
| `compiler-self`     | the compiler's own source (`mach build` at the repo root) — the realistic large workload |
| `ex-01-hello`       | `examples/01_hello` — minimal program, mostly stdlib        |
| `ex-09-collections` | `examples/09_collections` — a larger example exercising generics/collections |
| `synthetic`         | a generated standalone program (see `gen.sh`) for a scaling signal |

The synthetic input is generated by `bench/gen.sh [count]` (default 400
functions); set `BENCH_SYN_COUNT` to resize it. It is written to a throwaway
temp directory and removed on exit.

## Reading the output

```
input                 modules    lines  min(ms)  med(ms)
-----                 -------    -----  -------  -------
compiler-self             115    47511    14523    14529
ex-01-hello                15       18      235      235
ex-09-collections          21      109      421      421
synthetic                  13     3612     1313     1316
```

- `modules` — number of modules the build pulls in (including stdlib),
  reported by the compiler's verbose output.
- `lines` — total non-dependency `.mach` source lines in the input.
- `min(ms)` / `med(ms)` — fastest and median wall time across the timed runs.
  Prefer `min` as the cleanest signal (least noise from the OS scheduler);
  watch `med` for run-to-run stability.

A non-zero exit means at least one input failed to build; those rows show
`FAIL`.

## Artifacts

`mach build` always roots its output at the project being built, so each
build's objects and binary land in that project's `out/bench-tmp/`. The
harness removes those on exit and leaves every project tree as it found it —
no stray build artifacts. The official `out/bin/` (holding the compiler
itself) is never touched.

## Tracking regressions

Run the harness on a known-good build, save the table, and re-run after a
change. A meaningful jump in `min(ms)` for `compiler-self` (the largest, most
representative workload) is the primary regression signal. Keep the host
otherwise idle for comparable numbers; absolute timings are machine-specific.
